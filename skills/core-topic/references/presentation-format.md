# Presentation Format — Deep-Dive Output Guide

## Tone

**講義ではない。興奮した先輩エンジニアがコーヒー片手に語る感じ。**

- 「これ見て、ここがヤバいんだよ」という熱量
- 難しい概念でも身近な比喩で「あ、そういうことか」と思わせる
- 読み終わった後「自分もコード書きたい」と思わせるのがゴール
- 敬語は使わない。タメ口で。ただし下品にはならない
- 技術用語・コード・変数名は英語のまま

## Output Structure

```markdown
---

## 🔥 今日の Core Topic

**{title}**
{category} · {author} · {era}
{url}

> {one_line_hook — シードの hook をそのまま、または膨らませる}

---

## 📖 これは何か

{2-3段落で背景を説明}
{この PR/Issue/RFC が生まれた時代の状況}
{どんな問題を解決しようとしたのか}

---

## 🎯 ここがキモ

{核心部分の技術的解説}
{実際のコードスニペットや diff を引用}
{introspection markers を活用}

🤔 なぜこの設計になったのか
💡 Wizard ポイント — 一段上に行ける知識
⚡ パフォーマンス/エコシステムへの影響

---

## 🔬 コードを読む

{実際の実装から重要な部分を抜粋}
{行単位で何が起きているか解説}
{「ここが美しい」ポイントを示す}

```language
// 実際のコードスニペット
```

---

## 🌊 Before → After

{この変更の前後で何が変わったか}
{Before: こうだった（問題点）}
{After: こうなった（解決）}

---

## 💡 Wizard ポイント

`★ Insight ─────────────────────────────────────`
{この知識を持っていると何が違うか}
{実務でどう活きるか}
{関連する概念へのリンク}
`─────────────────────────────────────────────────`

---

## 🛠 手を動かしてみよう

**提案: {具体的な実験タスク}**

{utils または next-play で試せる小さな実験}
{所要時間目安: 30-60分}
{何を学べるか}

---
```

## Introspection Markers

解説中に自然に織り交ぜる。ブロックではなくインラインで。

| Marker | 用途 | 例 |
|--------|------|------|
| 🤔 | 設計判断の背景 | 🤔 なぜ requestIdleCallback ではなく MessageChannel を使ったのか |
| 🎯 | 核心部分の特定 | 🎯 ここがキモ。この1行で優先度が決まる |
| ⚡ | 性能/影響 | ⚡ これにより初回レンダリングが 40% 高速化した |
| 💡 | 上級知識 | 💡 これを知ってると面接で「おっ」と思われるレベル |
| 🔍 | 詳細追跡 | 🔍 この関数を追うと reconciler の核心に辿り着く |

## Length Guide

- 全体で 2000-4000 文字程度（読むのに 5-10 分）
- 長すぎると読む気が失せる。短すぎると物足りない
- コードスニペットは 10-30 行以内に絞る
- 「もっと知りたい」と思わせて終わるのがベスト

## Anti-Patterns

- 教科書的な網羅的説明（退屈）
- 「〜です」「〜ます」の敬体（距離感が出る）
- コード全体を貼り付けて「読んでください」（不親切）
- 著者の経歴を延々と紹介（本題に入れ）
- 「これは重要です」と言うだけで理由を示さない（空虚）
